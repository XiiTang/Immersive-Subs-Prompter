# Settings Panel Redesign Design

> **Status:** ✅ COMPLETED - Implementation verified with 103 tests passing.

## Goal

重构当前设置窗口的布局和 UI 风格，把它从“多个 section 和卡片堆叠起来的配置面板”整理成更接近桌面系统偏好设置的体验。

本次设计必须满足以下结果：

- 保留现有设置数据结构、读写链路和主要业务逻辑
- 重构设置窗口的信息架构，让顶层分区更清晰、更容易定位
- 重构视觉语言，减少“很多框框”的感受
- 让右侧内容区成为可连续阅读的长页，而不是杂乱堆叠的表单块
- 修正滚动区域体验，包括滚动条占位、右侧安全边距和章节定位
- 在常见桌面宽度下保持稳定、现代、统一的设置体验

## Non-Goals

本次设计不包含：

- 修改 settings 持久化协议、IPC 数据格式或 store 数据模型
- 顺手重写每个设置项的业务逻辑
- 引入多级路由或独立页面栈
- 为移动端或极窄宽度做单独适配
- 大规模重命名现有设置字段或重新定义设置功能

## Current Problems

当前设置面板的主要问题不是功能缺失，而是结构和视觉表达失衡：

- 顶层分区、章节、字段组之间的层级不够清晰，用户扫一眼很难建立整体结构
- 大量边框、卡片和容器叠加在一起，形成明显的“框套框”观感
- 内容节奏不稳定，有些区域过密，有些区域又缺少组织
- 右侧滚动区域没有明确的安全边距，视觉上容易贴边，也没有给滚动条预留稳定空间
- 不同类型的内容块缺少统一规则，普通表单、配置列表、规则编辑区的呈现语言不够一致
- 整体更像后台配置页面，而不是轻量、现代的桌面偏好设置界面

## Product Decision

采用 `Settings as Document` 方案。

具体含义是：

- 左侧保留稳定的一级导航，承担“目录”职责
- 右侧改成单一连续滚动的长页内容区
- 顶层分区在右侧都以大章节形式展开，而不是一次只显示一个分区页面
- 主要通过标题层级、留白、内容宽度控制和弱分隔组织界面
- 只有复杂模块保留轻量 surface，不再默认给所有区域套卡片

不采用“每次只显示一个分区页面”的方案，也不采用“继续保留大量卡片但换个皮肤”的方案。

原因：

- 连续长页更符合桌面偏好设置的阅读流，也更适合用户顺序浏览和逐项检查
- 左侧目录与右侧长页组合，能同时解决定位能力和整体感混乱的问题
- 视觉上的现代感应来自秩序和比例，而不是加更多装饰或更强烈的品牌化样式

## Design Principles

本次重构遵循以下原则：

- `Structure over containers`：优先用层级、留白和节奏表达结构，而不是依赖边框和卡片
- `Navigation as orientation`：左侧导航不仅是跳转器，也要持续提供当前位置感知
- `One visual system`：普通字段、列表行、操作按钮和说明文字必须属于同一套系统
- `Readable long page`：右侧长页必须像文档一样可读，不能只是把现有 section 纵向重排
- `Preserve logic, redesign presentation`：业务逻辑尽量复用，重点改展示结构和交互节奏

## Information Architecture

### Top-Level Sections

顶层分区收敛为以下 6 个一级章节：

- `General`
- `Profiles`
- `Rules`
- `Transcription`
- `Media Server`
- `Cache`

这些顶层分区只承担心智分类，不承担额外视觉复杂度。用户进入设置后，应能立刻理解“自己当前在哪个大类，以及其他大类在哪里”。

### Navigation Model

设置窗口采用稳定的左右双栏骨架：

- 左侧为固定宽度目录导航
- 右侧为单一滚动容器

左侧点击某个分区时，右侧平滑滚动到对应大章节，而不是切换成另一张完全不同的页面。右侧滚动时，左侧导航的选中状态要跟随当前章节更新。

这种模型的目标是同时保留：

- 长页阅读的连续性
- 一级分区的快速定位能力
- 用户在长内容中的位置感

### Section Composition

每个一级章节在右侧由三层组成：

1. 章节标题
2. 一句简短说明
3. 若干字段组或结构化内容块

章节内部不再默认使用“section 外层卡片 + 内层卡片 + 字段框”的多层嵌套结构。普通字段组直接顺着页面节奏展开，只有复杂区域才允许额外 surface。

## Layout Design

### Window Shell

整体窗口应维持现有独立设置窗口的大框架，但内部布局调整为更轻、更稳定的偏好设置壳层。

左栏应包含：

- 窗口标题或设置标题
- 简短的辅助说明
- 一级导航目录
- 可选的轻量底部信息区，例如版本信息

右栏应包含：

- 页面总标题或设置引导语
- 所有一级章节的连续内容

左栏是稳定目录，不随右侧内容高度变化。右栏是唯一主滚动区域，避免内部出现过多嵌套滚动。

### Content Width

右侧内容不能直接铺满整个滚动容器。需要采用“外层滚动容器 + 内层受控阅读宽度”的结构。

这样做的目的是：

- 保持字段扫描距离合理
- 避免超宽表单降低可读性
- 给右侧滚动条预留明确安全空间
- 让复杂区块和普通字段组都能在统一栅格下布局

### Responsive Boundaries

本次设计以桌面常见宽度为主，不做激进响应式变形。但即使在较窄窗口下，也必须保持：

- 左侧导航宽度稳定，不挤压到无法辨识
- 右侧内容不因滚动条出现而发生明显跳动
- 字段、说明文字和按钮不贴边、不重叠、不截断到不可用

## Visual Language

### Overall Tone

风格基调采用轻量系统感，而不是后台控制台感。

界面应接近桌面偏好设置的观感：

- 中性色为主
- 留白充足
- 低对比度但清晰的结构分层
- 装饰极少，强调秩序和阅读舒适度

不采用高饱和品牌色主导全页，也不依赖重阴影、强描边或大面积卡片堆叠制造层次。

### Surface Strategy

整体层次建议分成三层：

- 窗口背景层
- 内容承载层
- 少量高复杂度模块的轻量 surface

普通章节和字段组之间主要通过以下方式区分：

- 更大的垂直留白
- 标题层级差异
- 极浅分隔线
- 背景层次的轻微变化

只有以下类型区域允许使用更明确的 surface：

- 实体列表管理，如 `Profiles`
- 配置集合管理，如 `Media Server`
- 结构较复杂的编辑模块，如 `Rules`

即使使用 surface，也必须避免厚边框和明显“卡片堆叠”。

### Typography and Hierarchy

文本层级需要严格收敛：

- 页面标题层
- 章节标题层
- 字段标签层
- 说明文字层

每一层都应有固定的字号、字重和颜色，不允许同级文本在不同模块里表现出不同视觉语气。这样才能把长页从“多个组件拼起来”变成“同一份文档”。

### Color Usage

颜色策略应以中性色为主，强调色只用于：

- 左侧当前导航项
- 聚焦态
- 主操作按钮
- 状态反馈

强调色不应用于大面积背景，不应用于所有按钮，也不应把每个 section 都染成独立视觉块。

## Content Rhythm

### Long-Page Reading Flow

右侧长页必须具备明确的阅读节奏：

- 页面顶部先提供整体说明
- 一级章节之间拉开明显距离
- 每个章节内部先解释这个区块做什么，再展示字段
- 普通字段组以自然文档流平铺
- 复杂区域在视觉上被清楚提取，但仍服从整体节奏

用户滚动时应感受到的是“顺着读下去”，而不是“穿过一个又一个配置盒子”。

### Field Group Rhythm

普通字段组的推荐结构为：

- 字段组标题
- 一句说明
- 相关字段

字段标签优先放在控件上方，说明文字紧贴字段本体，不要漂在远处。开关类设置可以采用横向行布局，但整行高度、间距和对齐必须统一。

### Complex Areas

复杂区域不能按普通表单处理：

- `Profiles` 更适合“管理列表 + 当前对象详情编辑”的结构
- `Rules` 更适合“规则列表 + 当前规则编辑器”的结构
- `Media Server` 更适合“配置条目集合 + 当前连接配置详情”的结构

这些区域在逻辑上是对象集合管理，不是简单字段集合。设计上应允许它们保留更强的结构感，但必须与全局视觉系统统一。

## Scrolling and Positioning

### Scroll Container

右侧必须只有一个主滚动容器。避免在同一阅读区中再嵌套多个纵向滚动区域，否则会破坏位置感和滚动一致性。

### Scrollbar Space

必须为滚动条预留稳定占位，避免以下问题：

- 内容宽度在“有滚动条 / 无滚动条”之间跳动
- 右侧字段、按钮或说明文字紧贴滚动条
- 视觉上出现“内容被压在边缘”的感觉

右侧内容列需要明确的右边距和滚动条安全区，这是本次重构的硬要求。

### Orientation While Scrolling

滚动过程中应持续提供定位能力：

- 左侧导航根据当前章节高亮
- 点击左侧导航时右侧滚动到对应章节
- 右侧顶部可保留轻量 sticky 区域，帮助用户明确当前所在的大分区

sticky 区域必须轻，不应重新引入厚重标题栏或遮挡内容。

## Component Rules

为避免不同模块各自演化出不同风格，需要建立统一组件规则：

- 一个页面只保留一种主输入样式
- 输入框、下拉框、数字框和文本域共享圆角、边框、聚焦态和高度体系
- 主按钮、次按钮、危险按钮的优先级表现固定
- 列表行样式统一，不允许同类对象列表在不同章节里像不同产品
- 说明文字的颜色和密度固定，不允许有的区域说明过重、有的区域几乎不可见

这些规则是视觉统一的底线，不应在模块级组件里随意偏离。

## Section-Specific Guidance

### `General`

`General` 应作为设置页的入口章节，适合承载相对基础、全局性的偏好项。它的目标是低认知负担，因此应采用最轻的布局形式，避免复杂列表和强容器感。

### `Profiles`

`Profiles` 是高复杂度区域。允许保留轻量 surface 和对象管理结构，但外层仍应服从右侧长页的大章节节奏，不能变成另一个独立应用。

### `Rules`

`Rules` 需要比普通字段更强的结构化表达，因为它涉及列表、排序、编辑和校验。这里可以保留更清晰的编辑区边界，但应减少无意义装饰，把重点放在信息组织和操作清晰度上。

### `Transcription`

`Transcription` 多为参数型配置，适合采用分组清晰的文档式表单。它不应获得比内容本身更强的视觉存在感。

### `Media Server`

`Media Server` 与 `Profiles` 类似，也属于配置实体集合管理。重点是把“配置列表”和“当前配置详情”区分清楚，但避免和全局 shell 产生双重重型边框。

### `Cache`

`Cache` 和危险操作区应放在章节靠后位置，与普通设置保持适当距离。这里的重点是边界清晰、操作谨慎，而不是视觉上更醒目或更厚重。

## Accessibility and Usability

虽然本次重点是布局和视觉重构，但仍需满足基本可用性要求：

- 文本和控件保持足够对比度
- 焦点态明确可见
- 导航、字段和按钮的点击目标足够稳定
- 键盘导航不会因滚动和 sticky 行为失效
- 危险操作与普通操作在视觉优先级上有清楚区分

## Validation

本次重构完成后，至少应满足以下验收标准：

- 用户第一次打开设置页时，能快速理解顶层分区结构
- 在长页滚动过程中，用户始终知道自己位于哪个大章节
- 页面视觉上明显减少“框套框”的感觉
- 右侧内容在滚动条附近仍保有舒适边距，不出现贴边压迫感
- 普通表单区、对象列表区和复杂编辑区虽然结构不同，但视觉语言保持统一
- 窗口宽度变化到常见较小桌面尺寸时，布局仍然稳定可读
- 所有现有设置功能继续可用，不因重构展示层而失效

## Testing Strategy

测试重点应放在“布局和交互回归”，而不是业务逻辑新增：

- 验证左侧导航与右侧章节定位、高亮联动正确
- 验证长页滚动下的 sticky 区域不会遮挡内容
- 验证有无滚动条时右侧内容宽度稳定
- 验证复杂区块和普通表单区在视觉上属于同一系统
- 验证常见窗口宽度下的阅读宽度、边距和按钮布局稳定
- 验证现有设置修改、保存和同步行为没有回归

## Implementation Notes

本次实现应优先重构以下层面：

- 设置窗口外层 shell 与内容栅格
- 左侧导航和右侧章节映射关系
- 全局 spacing、surface、typography 和 form control 样式规则
- `Profiles`、`Rules`、`Media Server` 等复杂区域与新 shell 的整合方式

应尽量复用现有设置 store、IPC 和字段更新逻辑，把风险控制在 renderer 布局和样式层。
