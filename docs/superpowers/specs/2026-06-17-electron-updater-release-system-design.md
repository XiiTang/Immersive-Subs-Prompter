# Electron Updater Release System Design

## Goal

Immersive Subs Prompter uses a real desktop auto-update system. The desktop app checks for updates, downloads the selected updater asset, reports progress, and installs the downloaded update after the user chooses to restart.

The final desktop update system is `electron-builder` plus `electron-updater`. It does not use the app-side `releases/latest.json` static manifest design. The renderer never builds release URLs, never downloads installers directly, and never opens arbitrary release links.

The browser extension remains a release artifact for Chrome Web Store and Firefox AMO submission. Extension store updates are not handled by the desktop updater.

## User Constraints

This design describes only the final product shape. It does not describe implementation phases, migration steps, or temporary intermediate behavior.

Cherry Studio's final updater shape is:

- desktop packages are built by `electron-builder`
- release assets include platform installers, updater YAML metadata, and blockmap files
- the main process owns `electron-updater`
- update events are broadcast from main to renderer
- `autoUpdater.checkForUpdates()` performs detection
- `autoUpdater.downloadUpdate()` performs download
- `autoUpdater.quitAndInstall()` performs installation
- `autoInstallOnAppQuit` is disabled so installation requires an explicit user action
- `dev-app-update.yml` supports updater testing outside production releases

Cherry Studio also has channel, mirror, and compatibility-routing logic through `app-upgrade-config.json`. Immersive Subs Prompter does not include that extra routing in the final shape. This product has one stable desktop update channel backed by GitHub Releases.

## Product Version Model

The repository has one product version per release.

These package versions must match:

```text
package.json
apps/desktop-app/package.json
apps/extension/package.json
```

Release tags use:

```text
vX.Y.Z
```

The desktop updater compares the packaged app version with the latest release reported by `electron-updater`. Browser extension ZIP filenames also use the same product version.

## Desktop Packaging

The desktop app release package is built with `electron-builder`.

Final desktop packaging characteristics:

- `apps/desktop-app` has an `electron-builder` configuration as the only desktop release packaging contract.
- The app keeps the existing built output contract: main, preload, and renderer bundles are produced before packaging.
- The packaged app identity uses the existing product name, bundle identifier, executable name, icons, resources, and fuse hardening expectations.
- Release artifact filenames use the stable `Immersive-Subs-Prompter` slug so updater metadata and uploaded assets use the same names.
- Native and runtime resources that must exist on disk are placed in `asarUnpack` or `extraResources`.
- Electron Forge makers, Forge release collection scripts, and Forge-only output assumptions are not part of the final release system.

Target artifacts:

- macOS: DMG for user installation and ZIP for updater delivery
- Windows: NSIS installer for user installation and updater delivery
- Linux: AppImage for updater delivery, plus deb and rpm as manual distribution artifacts

The updater install path is first-class for macOS ZIP, Windows NSIS, and Linux AppImage. Deb and rpm are release downloads, not in-app updater installation targets.

## Update Feed

Production updates use GitHub Releases for:

```text
XiiTang/Immersive-Subs-Prompter
```

`electron-builder` publishes updater metadata expected by `electron-updater`, including:

- `latest.yml`
- `latest-mac.yml`
- `latest-linux.yml`
- blockmap files where generated by the platform target
- platform installers and archives referenced by the YAML metadata

The desktop app does not fetch:

```text
https://raw.githubusercontent.com/XiiTang/Immersive-Subs-Prompter/main/releases/latest.json
```

There is no app-side static release manifest. The update feed is the builder-generated updater metadata hosted with GitHub Release assets.

The release artifact check verifies every `url` and `path` reference in the builder-generated updater metadata points to a desktop asset uploaded with the release.

Local updater testing uses a `dev-app-update.yml` equivalent that points `electron-updater` at an explicit test feed. Production code does not infer update feeds from renderer state.

## Main-Process Update Service

The main process owns the updater service.

Responsibilities:

- configure `autoUpdater` request behavior and logger
- set `autoUpdater.autoDownload = false`
- set `autoUpdater.autoInstallOnAppQuit = false`
- call `autoUpdater.checkForUpdates()` for manual and automatic checks
- call `autoUpdater.downloadUpdate()` only when the user starts download
- call `autoUpdater.quitAndInstall(true, true)` only after an update is downloaded and the user chooses install
- translate `electron-updater` events into a stable app release state
- broadcast release state to the main window and settings window

The service exposes only explicit IPC commands:

- get current release state
- check for updates
- download the available update
- install the downloaded update

The renderer cannot pass a download URL, feed URL, filesystem path, or installer path into any release IPC handler.

## Release State

The app release state is explicit and renderer-safe.

States:

- `idle`
- `checking`
- `available`
- `unavailable`
- `downloading`
- `downloaded`
- `installing`
- `error`

State data:

- current version
- latest version when known
- update availability
- update info from `electron-updater`
- localized release notes when available
- check timestamp
- progress percentage
- transferred bytes
- total bytes
- bytes per second
- error code and message

Automatic checks run after startup and are limited to one check attempt per 24 hours. Manual checks always run immediately. Failed automatic checks still update the last check timestamp so a transient network failure does not retry constantly.

## Renderer Experience

The Settings update section is the user-facing updater surface.

Final controls:

- current version
- automatic update check switch
- "Check for updates"
- "Download update"
- progress bar while downloading
- downloaded version and release notes
- "Install and restart"
- non-fatal error status with a concise message

Button availability follows the main-process release state. The renderer does not maintain independent update truth and does not store release URLs.

When an update is downloaded, the app does not silently restart. The user explicitly installs and restarts.

## Release Automation

The release workflow builds one product release from a `vX.Y.Z` tag. Manual dispatch inputs must also use the `vX.Y.Z` tag shape.

Final workflow outputs:

- macOS desktop artifacts and updater metadata
- Windows desktop artifacts and updater metadata
- Linux desktop artifacts and updater metadata
- Chrome extension ZIP
- Firefox extension ZIP
- checksum file for published assets
- published GitHub Release containing all desktop and extension assets

The workflow validates:

- unified package versions
- release tag shape
- desktop app typecheck and tests
- extension build and tests
- generated updater metadata is present for each updater-supported platform
- release assets use expected names
- updater metadata references only release asset filenames that exist, not arbitrary absolute URLs
- macOS release signing and notarization secrets are configured before macOS packaging starts

The workflow does not open a pull request for `releases/latest.json`. That file is not part of the final desktop update system.

## Signing And Install Trust

The updater path is designed around platform-native installation trust.

Final expectations:

- macOS release artifacts are signed and notarized for updater-compatible installation.
- Windows NSIS artifacts use the configured signing path when signing credentials are present.
- Linux AppImage update delivery relies on the AppImage updater path produced by `electron-builder`.
- Release metadata does not claim an artifact is signed when it is not.

Unsigned local builds may be used for development feed testing. Production macOS releases fail before packaging if signing or notarization secrets are missing.

## Error Handling

Updater errors are non-fatal to core subtitle functionality.

Handled error classes:

- feed unavailable
- no update available
- updater metadata invalid or incomplete
- unsupported platform updater target
- download failure
- install failure

Errors are reported in the update settings surface and logged in the main process. The app does not fall back to parsing GitHub pages, raw JSON manifests, or renderer-provided download URLs.

## Security Boundary

The updater security model is main-process-only.

Rules:

- renderer IPC cannot carry URLs into the updater
- renderer IPC cannot carry installer paths into the updater
- generic external URL openers are not introduced for update actions
- release state sent to the renderer is display data, not authority
- updater feed configuration is packaged main-process configuration
- update installation can only happen after the main process has observed an `update-downloaded` event

Existing security intent around release URL control remains part of the final contract, but the final action is updater download and installation rather than opening a download page.

## Tests

Main-process tests cover:

- check success with update available
- check success with no update
- check error
- download progress state
- downloaded state
- install command requires downloaded state
- automatic check rate limiting
- renderer-supplied URL and path rejection at the IPC surface

Renderer tests cover:

- idle settings UI
- checking state
- available state
- downloading state with progress
- downloaded state with install action
- error state
- no renderer URL forwarding

Release script and workflow tests cover:

- unified versions
- expected builder artifact names
- expected updater metadata files
- extension ZIP outputs
- absence of `releases/latest.json` in the update flow

## Out Of Scope

The final system does not include:

- `releases/latest.json` update checks
- Electron Forge desktop release packaging
- renderer-controlled download URLs
- old release manifest compatibility
- beta or rc update channels
- region-based mirrors
- staged or segmented upgrades
- mandatory updates
- silent background install on app quit
- extension auto-update through the desktop app
