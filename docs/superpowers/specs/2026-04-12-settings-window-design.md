# Settings Window Design

## Goal

将当前嵌在字幕主窗口里的设置面板彻底单拎出来，改造成一个独立的设置窗口，并重构为“左侧导航栏 + 右侧主内容”的固定尺寸布局。

本次设计必须满足以下结果：

- 点击主窗口右上角设置按钮时，打开独立设置窗口，而不是在主窗口内展开设置面板
- 设置窗口为单例窗口，重复点击设置按钮时只聚焦到该窗口，不做显示/隐藏切换
- 设置窗口固定尺寸、不可缩放，但允许拖动整个窗口
- 主字幕窗口在设置窗口打开期间继续可用，不被禁用、不被替换
- 设置窗口关闭后，下次打开按默认位置和默认尺寸显示，不恢复上次位置
- 设置内容整体重构为左侧一级导航、右侧单页内容区
- 现有设置数据读写链路继续复用，不在本次重构中改造 settings 数据协议

## Non-Goals

本次设计不包含：

- 改造设置数据结构、IPC 数据格式或持久化协议
- 为设置窗口增加多实例能力
- 记住设置窗口上一次关闭时的位置或大小
- 为固定尺寸窗口额外实现可缩放布局分支
- 顺手重构与设置无关的字幕主窗口、自动隐藏或播放逻辑
- 引入复杂的窗口路由系统或深层 URL 状态管理

## Current Problems

当前实现存在以下结构性问题：

- 设置面板由主窗口 renderer 通过 `store.isSettingsOpen` 控制显示，本质上仍是主界面里的一个局部视图，而不是独立工具窗口
- `App.vue` 同时承载字幕主视图和设置面板，导致窗口职责混杂
- 当前设置布局是多个 section 的堆叠长页，不适合固定尺寸的桌面设置窗口
- `Profiles`、`Transcription`、`Media Server` 已经各自带有内部侧栏或列表编辑结构，和“全部堆在一个滚动页里”的外层结构存在冲突
- 主进程只有一个主窗口管理模型，没有明确的设置窗口管理边界
- “设置是否打开”被建模为 renderer 展示态，而不是主进程窗口态，不利于后续独立窗口扩展

## Product Decision

采用“独立设置窗口 + 固定双栏壳层”的方案。

具体含义是：

- 主窗口只负责字幕相关界面
- 设置由主进程创建第二个 `BrowserWindow` 承载
- 设置窗口内部采用稳定的左侧一级导航 + 右侧当前分类内容布局
- 现有各类设置模块尽量复用已有表单与编辑逻辑，但在新窗口壳层下重新组织页面结构

不采用“继续在主窗口里做抽屉/覆盖层”的方案，也不采用“窗口内全量路由化多页面系统”的激进改造。

原因：

- 独立窗口能真正解耦主窗口职责
- 固定尺寸前提下，双栏布局比长页更适合桌面设置场景
- 保留现有设置业务逻辑，能把风险控制在窗口边界和页面壳层重组上

## Window Architecture

### Window Roles

窗口职责拆成两类：

- `MainWindowManager`
  - 管理字幕主窗口
  - 不再负责设置面板显示
- `SettingsWindowManager`
  - 管理独立设置窗口的创建、聚焦、关闭和引用清理

设置窗口是主进程层面的单例窗口，不是主窗口里的某个 Vue 状态分支。

### Open Behavior

主窗口点击设置按钮时：

- 如果设置窗口不存在，则创建并显示
- 如果设置窗口已存在但被最小化，则恢复并聚焦
- 如果设置窗口已存在且可见，则直接聚焦到前台

不采用 toggle hide 语义。

原因：

- 按钮语义是“打开设置”，不是“切换抽屉”
- 重复点击聚焦更符合独立设置窗口的用户预期
- 可以避免误隐藏后用户误以为窗口丢失

### Close Behavior

关闭设置窗口时：

- 只关闭设置窗口本身
- 主字幕窗口继续保留并可操作
- 主进程清理设置窗口引用，便于下次重新创建

关闭后重新打开：

- 使用默认尺寸
- 使用默认位置
- 默认进入 `General` 导航页

不恢复上次位置，也不持久化上次导航页。

### Window Constraints

设置窗口具备以下约束：

- 固定尺寸
- 不允许用户 resize
- 允许拖动整个窗口
- 不允许全屏
- 保持单实例

拖拽仅由明确的标题栏区域提供，按钮、输入框、导航项和表单控件不参与拖拽命中。

## Renderer Separation

### Main Window Renderer

主窗口 renderer 只保留字幕相关 UI：

- `HeaderBar`
- `SubtitleView`
- 与字幕展示相关的状态和样式

主窗口 renderer 不再渲染 `SettingsPanel`。

原先通过 `store.isSettingsOpen` 控制内嵌设置面板显示的流程应被移除或降级为过渡实现，主流程改为调用新的窗口 IPC，例如：

- `openSettingsWindow`

### Settings Window Renderer

设置窗口 renderer 作为独立入口：

- 拥有自己的根组件，例如 `SettingsWindow` 或 `SettingsApp`
- 负责设置窗口标题栏与拖拽区
- 负责左侧导航状态
- 负责右侧当前页面切换

它与主窗口共享同一套设置数据来源，但不共享同一个组件树。

### Shared Settings Data

设置数据读写链路继续沿用现有 store / preload / IPC 管道：

- 读取当前 settings
- 更新 global / profiles / rules / transcription / media server / cache
- 让主窗口继续实时响应设置变化

本次重构只改变窗口承载方式和页面组织方式，不改变 settings 协议本身。

## Settings Layout

### Top-Level Navigation

设置窗口采用固定两栏布局：

- 左栏：一级导航，固定宽度
- 右栏：主内容区，只显示当前分类内容

一级导航项为：

- `General`
- `Profiles`
- `Rules`
- `Transcription`
- `Media Server`
- `Cache`

该导航不做折叠式响应式适配，直接围绕固定尺寸窗口设计。

### Content Strategy

右侧内容区一次只展示一个大类页面，不再把所有设置 section 堆成一整页。

这样做的原因：

- 固定尺寸窗口适合稳定信息架构，不适合继续使用长滚动页
- 用户进入设置时通常带着明确目标，按分类切换比通读长页更高效
- 现有复杂模块本身已经具备对象列表与编辑器结构，更适合作为独立页面承载

## Page Design

### `General`

`General` 作为应用级总览页，承载：

- 语言
- 开机自启
- 网络与连接设置
- 快捷键
- 进程黑名单

它应采用单页表单或分组卡片，但只在当前页面内组织，不再与其他大类并列堆叠。

### `Profiles`

`Profiles` 保留“列表 + 编辑器”的对象管理结构：

- 左侧内栏为 profile 列表和增删复制动作
- 右侧为当前 profile 详情编辑器

原因：

- profile 本质上是可切换实体集合
- 压平成单页表单会降低多 profile 管理效率

### `Rules`

`Rules` 从“上方列表 + 下方表单”调整为“列表 + 编辑器”双栏结构：

- 左侧显示规则列表与排序状态
- 右侧显示新增/编辑表单

原因：

- 规则编辑属于典型对象选中后编辑的模式
- 双栏结构更适合固定尺寸窗口，不会在垂直空间内频繁挤压

### `Transcription`

`Transcription` 继续保留 config 列表 + 详情编辑的总体结构，但需要收敛当前内容层次：

- 左侧内栏显示 transcription config 列表
- 右侧编辑器显示 provider、模型、路径、下载状态和运行参数

该页面允许保留内部卡片，但必须围绕固定尺寸右栏收敛为可控的滚动区域。

### `Media Server`

`Media Server` 延续现有 server list + editor 模式：

- 左侧为 server 列表
- 右侧为当前 server 详情配置

该模块无需改动核心编辑语义，只需适配新窗口壳层。

### `Cache`

`Cache` 保持为单页表单：

- 开关
- 缓存路径
- 保留时间
- 统计信息

该页不需要额外二级导航。

## Component Boundaries

### Shell Components

建议新增设置窗口壳层组件，职责明确分离：

- `SettingsWindowShell`
  - 负责标题栏、窗口容器、拖拽区域
- `SettingsNav`
  - 负责左侧一级导航
- `SettingsContent`
  - 负责根据当前导航项切换页面

这些壳层组件只负责页面结构和导航，不负责 settings 数据业务细节。

### Page Components

现有设置模块重组为页面级组件：

- `SettingsGlobal`
- `SettingsProfiles`
- `SettingsRules`
- `SettingsTranscription`
- `SettingsMediaServer`
- `SettingsCache`

这些页面级组件继续持有各自设置项的编辑逻辑。

对于明显依赖旧长页布局的模块，可以在本次重构中定向调整，但不需要为了“纯粹的抽象完美”而重写全部内部逻辑。

### Deprecated Boundary

`SettingsPanel.vue` 作为“主窗口中的内嵌设置壳层”将不再是长期架构核心。

允许短期过渡，但最终应退出主流程。

原因：

- 它绑定了旧模型：设置是主窗口内部状态
- 新设计下，设置应该由独立窗口 renderer 承载

## IPC And Window State

### New IPC Surface

主窗口和主进程之间需要新增明确的窗口控制语义：

- `open settings window`
- 可选：`close settings window`

不建议继续沿用“toggle settings panel”之类旧命名，因为这会把旧内嵌面板模型继续带入新设计。

### State Ownership

状态所有权明确如下：

- 主进程拥有“设置窗口是否存在、是否可聚焦”的窗口状态
- 设置 renderer 拥有“当前导航项”的页面状态
- 共享 store 拥有“具体设置数据”的业务状态

原 `store.isSettingsOpen` 这类内嵌展示态不再作为主流程控制状态。

## Error Handling And Edge Cases

需要明确处理以下边界情况：

- 设置窗口尚未创建时，打开请求应正常创建窗口，而不是静默失败
- 设置窗口对象仍存在但处于最小化状态时，应恢复并聚焦
- 设置窗口被其他应用遮挡时，重复点击设置按钮应把它带到前台
- 设置数据尚未完成初始化时，右侧内容区应显示 loading 或 skeleton，而不是空白闪烁
- 某个设置项更新失败时，沿用现有 store / IPC 错误处理方式，不在本次重构中额外发明新的全局错误机制
- 主窗口与设置窗口同时存在时，设置变更应仍然即时反映到主窗口表现上

## Testing Strategy

验证范围聚焦在窗口边界与核心导航行为，不把测试面铺得过大。

### 1. Main Process Window Behavior

至少验证：

- 打开设置时会创建独立设置窗口
- 重复打开不会创建第二个设置窗口
- 设置窗口已存在时会执行聚焦逻辑
- 设置窗口为固定尺寸且不可缩放

### 2. Renderer Structure

至少验证：

- 主窗口不再渲染旧 `SettingsPanel`
- 设置窗口根组件能渲染左侧导航和默认页面
- 点击导航项能切换右侧页面

### 3. Page Regression

至少覆盖高复杂度页面的基本回归：

- `Profiles`
- `Rules`
- `Transcription`

这些测试重点确认：

- 页面在新壳层下能正常渲染
- 基本交互仍然会触发 store 更新
- 页面切换不会破坏原有编辑流程

## Rollout Strategy

建议按以下顺序实现：

1. 在主进程引入独立设置窗口管理器与对应 IPC
2. 移除主窗口中的内嵌设置入口渲染逻辑
3. 建立设置窗口独立 renderer 入口与双栏壳层
4. 将现有六类设置模块迁入新导航体系
5. 对 `Rules` 等明显不适合长页结构的页面做针对性重排
6. 补齐窗口行为与导航切换测试

这个顺序的核心是先拆清窗口边界，再重组页面结构，最后做页面细化。

## Decision Summary

本次设计的关键决策如下：

- 设置从主窗口内嵌面板升级为独立窗口
- 设置窗口是固定尺寸、不可缩放、可拖拽的单例窗口
- 再次点击设置按钮时只聚焦，不隐藏
- 关闭后重新打开不恢复上次位置
- 设置窗口采用左侧一级导航、右侧单页内容区
- 保留现有 settings 数据协议与主要编辑逻辑，重点重构窗口壳层和页面组织方式
